## 2. 아키텍처 제약사항

### 기술적 제약
- 법제처 Open API 무료 티어 호출 한도 내에서 동작
- 로컬 벡터 저장소(numpy+JSON)로 사기유형·대응절차 큐레이션 데이터 관리
- LLM: Groq API 무료 티어(Llama 3.3 70B) — 요청 속도 제한(rate limit) 내에서 동작 (데모 트래픽
  수준에서 충분할 것으로 판단). Gemini는 계정 단위로 무료 티어 한도가 0으로 잡히는 문제가 확인돼
  Groq로 전환. GCP GPU VM은 신규 계정 GPU 쿼터 승인 리스크로 보류

### 조직적 제약
- 1인 개발 — 운영 복잡도 최소화, 새 프레임워크 도입 지양, 기존 검증된 패턴(3-Agent 구조) 재사용 우선

### 예산/인프라 제약
- 무료/저비용 스택 우선 (법제처 API 무료, 로컬 벡터 저장소 무료). Groq 무료 티어 한도 내에서만 사용
  가능 — 대회 제출 기간(약 5일)만 상시 가동하면 되므로 단기 호스팅 비용으로 제한 가능

## 3. 주요 아키텍처 결정 (초안)

> 상세 결정 기록은 `docs/design-docs/adr/` 에 ADR로 작성합니다.

| 결정 영역 | 선택 | 근거 |
|-----------|------|------|
| 레이어 구조 | 3-Agent 파이프라인 (Retrieval → Analysis → Feedback) | 검증된 아키텍처 패턴, 관심사 분리로 유지보수 용이 |
| 법적 근거 소스 | 법제처 Open API (실시간) | 공식 무료 API, 최신 법령 반영 |
| 실전 대응절차 소스 | 수작업 큐레이션 + 로컬 벡터 저장소 | 법령 조문에는 없는 실무 절차(신고 전화번호 등) 보완 필요 |
| 환각 방지 | 인용 검증 + 재분석 루프 | 제공되지 않은 법령 인용 시 자동 재분석하여 신뢰도 확보 |
| 판정 방식 | 단정적 판정 금지, 위험도+안전행동 병행 출력 | 오판 리스크(가족을 사기로 오인 등) 최소화 |
| UI | Streamlit | 빠른 프로토타이핑, 데모 적합 |

## 4. 리스크

| 리스크 | 영향도 | 발생 가능성 | 완화 전략 |
|--------|--------|-------------|-----------|
| Groq 무료 티어 요청 한도 초과로 데모 중 응답 실패 | Medium | Low | 데모 트래픽은 저부하 예상, 초과 시 대기·재시도로 대응 (유료 전환 금지) |
| 사기 대응절차 데이터가 API로 확보 안 되어 수작업 큐레이션 필요 (확인 완료 — API 없음) | Medium | 확정 | 경찰청/금감원 공식 자료 기반 초기 시드 10~20건으로 시작, 이후 확장 |
| 이전 대회 제출작(JB_LLM)과의 유사성으로 "결과물 재이용" 의심 | High | Low(주제·타깃·프롬프트 전면 재작성 시) | CLAUDE.md 프로젝트 맞춤 규칙에 명시된 재사용 제한 준수 |
| 사기 위험도 판정 오류로 사용자에게 잘못된 안심/불안 유발 | High | Medium | 단정 판정 금지 원칙 + 항상 안전 행동요령 병행 |

## 5. 검증 기준

아키텍처가 이 ARD를 만족하는지 확인하는 방법:

- [ ] 환각 방지 검증 — 미제공 법령 인용 시 재분석 루프가 정상 동작하는지 테스트
- [ ] 위험도 출력이 항상 "단정 아님 + 안전 행동요령" 형식을 따르는지 확인
- [ ] 배포된 URL이 제출 기간 상시 접근 가능한지 확인
- [ ] 주요 결정사항 ADR 작성 완료 (`docs/design-docs/adr/`)
